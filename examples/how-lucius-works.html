how lucius works

The pairing loop, explained by an artifact that traveled it. This page was written by Claude, pushed over the local API, versioned in SQLite, and is rendering in the canvas you're reading it on.

THE WHOLE THING IN ONE LINE

You talk; Claude ships a page; every pass is kept

STAGE 1 · YOU TALK

Steering is conversation, not configuration

You describe what you want — by voice mid-pairing, in chat, or by pasting a zoomed screenshot of the exact pixels that bother you. Real steering from the session that built this app:

"give me your options in lucius canvas as v4" · "no scroll tabs on top, i want layered sidebar" · "their height is not same" · "should be highlight based like tdoc"

Each of those became the next version of something within minutes. The unit of feedback is a sentence, not a ticket.

STAGE 2 · CLAUDE BUILDS

Grounded first, designed second, then rendered

The skill that drives this canvas has two hard layers. Truth rules: if the artifact describes a real system, Claude scans the actual repo first and simplifies vocabulary, never the system — omitting a real component is a defect. Taste rules: a strictly monochrome house style — ink is the only accent — typography-led hierarchy, tight radii, and an explicit anti-slop list. The artifact arrives as one self-contained HTML file:

lucius render page.html "label" — or the MCP render tool, from any Claude session.

STAGE 3 · EVERY PUSH IS A VERSION

Iterations are kept, never overwritten

The sidebar is a timeline per session. Scrub back any time — "wait, go back to the earlier one" is a click. The status pill inverts when you're behind and snaps you to the newest push. Sessions keep parallel workstreams apart; other Claude sessions can feed their own.

STAGE 4 · POINT & COMMENT

Feedback lands on the exact thing you mean

Highlight any text on this page — a Comment pill appears, and your note anchors to that exact phrase (the anchor carries the text plus context, so it survives new versions when the phrase does). Anchored comments render as underlined highlights; click one for its thread. The crosshair in the header is the other pointer: select an element and Claude reads exactly what you're looking at via get_selection.

Try it now on this sentence — this page is a live canvas, not a picture of one.

STAGE 5 · PUBLISH, INVITE, REACT

One click puts it on your own Cloudflare

The cloud button publishes every version of a session to your worker (provisioned once by lucius setup — your account, your R2 bucket, your URL). Docs default to link-visibility; flip one private and invite GitHub usernames — uninvited visitors hit a sign-in gate. Viewers highlight and comment on the published page; their comments flow back into this app within a minute, badge the header, and Claude replies on the page with what it changed: lucius reply <id> "fixed in v5" applied.
